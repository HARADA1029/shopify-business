# 担当エージェント案

本プロジェクトで活用する Claude Code エージェントの役割分担。
0→1 の立ち上げ段階のため、各エージェントはまず「設計・方針決定・検証」を担い、
実装は設計が固まってから段階的に進める。

---

## 0. project-orchestrator（全体統括）

**担当領域:** プロジェクト全体の進行管理、エージェント間の調整、意思決定の記録

### 設計段階（現在）
- プロジェクト計画の策定と更新
- エージェント間の依存関係と優先順位の管理
- 未決事項の洗い出しと意思決定の促進
- CLAUDE.md の維持・更新
- リスク一覧の管理と対策状況の追跡

### 実装段階
- マイルストーンの進捗管理
- エージェント間のブロッカー解消
- テスト計画の統括
- ローンチ判定基準の管理

**重要度:** 最高（常時稼働）

---

## 1. store-setup（ストア構築エージェント）

**担当領域:** Shopify ストアの設計・初期構築・基本設定

### 設計段階（現在）
- Shopify プラン比較と選定案の作成
- テーマ候補の比較・選定基準の整理
- ストア構造の設計（コレクション階層、ナビゲーション）
- 必須ページの構成案（About, Shipping Policy, Return Policy, FAQ）
- 必要アプリの調査・比較表の作成
- ドメイン名候補の整理
- 通貨・言語・税金の設定方針

### 実装段階
- テーマのインストールとカスタマイズ
- 基本設定の投入（通貨、配送ゾーン、税金）
- 必須ページの作成
- アプリのインストールと設定
- テスト注文による動作検証

**重要度:** 最高（立ち上げの中核。他のエージェントの成果物がここに集約される）

---

## 2. channel-sync-strategy（チャネル連携・在庫同期設計エージェント）

**担当領域:** eBay ↔ Shopify の連携方式・在庫同期方針・二重販売防止策の設計

### 設計段階（現在）
- 在庫同期の方式比較（リアルタイム API / 定期バッチ / 既存 SaaS）
- 既存 SaaS の調査（LitCommerce, Codisto, Sellbrite 等）
- 二重販売防止のアーキテクチャ設計
- 同期障害時のリカバリ方針の策定
- 在庫データの正規化ルール（eBay SKU → Shopify SKU のマッピング）
- 監視・アラート要件の定義

### 実装段階
- 選定した同期方式の実装またはサービス導入
- eBay API / Shopify API の連携コード
- 同期エラーハンドリングとリカバリ処理
- 監視ダッシュボードとアラート設定
- 在庫同期の負荷テスト・障害テスト

**重要度:** 最高（二重販売は信用とコストの両面で致命的）

---

## 3. catalog-migration-planner（商品移行設計エージェント）

**担当領域:** eBay 商品データの Shopify 移行に関する計画・ルール・検証

### 設計段階（現在）
- 移行対象 SKU の選定基準（売上ランク、カテゴリ、状態）
- eBay 商品データの構造分析（タイトル、説明文、画像、属性）
- カテゴリマッピングの定義（eBay カテゴリ → Shopify コレクション）
- 商品タイトル・説明文の変換ルール（eBay 形式 → Shopify 形式）
- 画像移行方針（eBay CDN → Shopify / 再アップロード手順）
- バリデーションルールの設計（移行後データの品質チェック項目）
- 初期 300〜500 SKU のリスト作成基準

### 実装段階
- 移行スクリプトの開発
- カテゴリ・属性マッピングファイルの作成
- テスト移行の実行と結果検証
- 移行データのバリデーション実行
- 本番移行の実行と確認

**重要度:** 高（移行データの品質がストアの第一印象を決める）

---

## 4. fulfillment-ops（出荷・配送・返品運用エージェント）

**担当領域:** 国際配送の設定、出荷フロー構築、返品対応、関税案内

### 設計段階（現在）
- 配送料金テーブルの設計（国別・重量別・サイズ別）
- 配送業者の比較（日本郵便 / FedEx / DHL / eBay 既存業者との兼用）
- 出荷フローの文書化（受注→ピッキング→梱包→発送→追跡番号連携）
- 返品ポリシーの策定（条件、送料負担、返金方法）
- 関税・禁制品の国別注意事項リスト
- 梱包資材と梱包基準の整理
- eBay 既存の出荷フローとの共通化・差分整理

### 実装段階
- Shopify の配送料金設定
- 追跡番号の自動連携設定
- 返品処理フローの構築
- 配送関連ページの作成（Shipping Policy, FAQ への反映）

**重要度:** 中〜高（eBay で既にノウハウがあるため設計は効率的に進められる）

---

## 5. growth-foundation（集客基盤設計エージェント）

**担当領域:** SEO 基盤、SNS 導線、メール取得導線など集客の土台設計

### 設計段階（現在）
- 商品ページの SEO 方針（メタタグ、構造化データ、URL 設計）
- SNS アカウント設計（Instagram, X）と投稿方針
- メールアドレス取得導線の設計（ポップアップ、フッター、初回割引）
- eBay 既存顧客への Shopify 誘導施策（サンクスカード、メール案内）
- Google Shopping 連携の要件整理
- 競合ストアの調査・ベンチマーク

### 実装段階
- SEO 設定の投入
- SNS アカウント開設と初期コンテンツ作成
- メール配信ツールの選定・導入（Shopify Email / Klaviyo）
- Google Shopping フィードの設定
- 広告の初期テスト（Google Ads / Meta Ads）

**重要度:** 中（立ち上げ初期はオーガニック + eBay 既存顧客が中心。基盤だけ先に設計）

## 6. price-auditor（価格監査エージェント）

**担当領域:** eBay → Shopify の価格変換の正当性検証、要確認商品の抽出、原田への確認リスト作成

**背景:**
eBay 出品価格には米国向け想定関税を上乗せした商品が混在しており、
一律 `× 0.91` の簡易ルールをそのまま適用すると二重取りや過小マージンのリスクがある。
price-auditor は移行候補を「そのまま変換して良い商品」と「原田の判断が必要な商品」に
仕分けする監査役を担う。

**参照:** `docs/shopify-price-strategy.md`

### 設計段階（現在）

- 価格分類ルールの定義（分類 A / B / C の判定条件）
  - A: 関税上乗せなし・$1,000 未満 → `× 0.91` で一括変換
  - B: 関税上乗せ済み → 初期移行から除外（原田に確認リスト提出）
  - C: $1,000 以上 → 手動レビュー対象（原田に確認リスト提出）
- 関税上乗せ済み商品の検出ヒューリスティック設計
  （同一カテゴリ内での価格外れ値、マージン率の偏差など）
- 要確認商品リスト（CSV + サマリ）のフォーマット定義
- 確認後の価格見直しフローの整理

### 実装段階

- 移行候補に対する価格監査スクリプトの開発
- 分類 A / B / C の自動仕分け実行
- 要確認リストの自動生成
- 原田の判断結果を反映する価格更新スクリプト
- 価格変換後のバリデーション（マージン率チェック、eBay との乖離チェック）

### 原田への通知フロー

```
price-auditor が移行候補を監査
        │
        ├── 分類 A: 自動変換 OK → そのまま移行パイプラインへ
        │
        ├── 分類 B: 関税上乗せの疑いあり
        │     → 確認リスト（CSV）を生成
        │     → 原田に提出:「この商品群は eBay 価格に関税が
        │       含まれている可能性があります。確認してください」
        │     → 原田の判断:
        │         (a) 上乗せあり → 初期移行から除外 or 個別価格設定
        │         (b) 上乗せなし → 分類 A として変換
        │
        └── 分類 C: 高額品（$1,000+）
              → 確認リスト（CSV）を生成
              → 原田に提出:「高額品です。Shopify 価格を
                個別に確認してください」
              → 原田の判断: 個別に Shopify 価格を指定
```

### 確認後の価格見直しフロー

```
1. 原田が確認リストに判断結果を記入（CSV の列に記入）
     │
2. price-auditor が判断結果を読み込む
     │
3. 判断に基づいて処理を分岐
     ├── 「変換 OK」  → 分類 A として × 0.91 適用
     ├── 「除外」     → 移行候補から除外
     └── 「個別価格」 → 原田指定の価格を Shopify 基準価格に設定
     │
4. 最終価格リストを生成 → catalog-migration-planner に引き渡し
```

### 他エージェントとの境界

| 関連エージェント | 境界 |
|---|---|
| **catalog-migration-planner** | price-auditor は「価格の正当性検証と仕分け」のみ。商品データの移行処理自体は catalog-migration-planner が担当。price-auditor の出力（最終価格リスト）を catalog-migration-planner が受け取る |
| **fulfillment-ops** | 送料テーブルの設計は fulfillment-ops が担当。price-auditor は送料を除いた商品価格のみを扱う |
| **store-setup** | Shopify の価格フィールド設定（Price / Compare at price / Cost per item）の定義は store-setup。price-auditor はフィールドに入れる値を算出する |

**重要度:** 高（初期移行の価格品質に直結する。分類 A が大多数であれば運用負荷は抑えられる）

**独立検討メモ:**
価格再計算・最適化（セール設計、為替連動、競合価格追従など）の業務が増えた場合、
price-optimizer を分離する。現時点では price-auditor が監査 + 簡易変換を一括で担う。

## エージェント間の依存関係（立ち上げ初期）

```
        project-orchestrator（常時：全体統括）
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
store-setup  channel-sync  catalog-migration
（ストア設計）  （同期方針）    （移行計画）
    │           │           │
    │           ▼           │
    │    ※ 同期方式が決まらないと  │
    │      移行を開始できない     │
    │           │           │
    └─────┬─────┘           │
          ▼                 │
   fulfillment-ops ◄────────┘
   （出荷フロー）        │
          │              ▼
          │        price-auditor
          │        （価格監査）
          │              │
          ▼              ▼
       growth-foundation
       （集客基盤）
```

**依存関係のポイント:**
- store-setup / channel-sync / catalog-migration は並行して設計を進められる
- ただし catalog-migration の**実行**は channel-sync の方式決定後に行う
- fulfillment-ops は store-setup と catalog-migration の設計に依存する（配送対象商品の特性を知る必要がある）
- price-auditor は catalog-migration の移行候補リストを入力として受け取り、価格監査済みリストを返す
- growth-foundation はストアの骨格ができてから本格始動する

## 最初に動かすべきエージェントの順番

### 第1陣（Day 1〜3: 同時並行で設計開始）

| 順位 | エージェント | 理由 |
|:---:|---|---|
| 1 | **project-orchestrator** | 全体計画の策定。他の全エージェントの前提になる |
| 2 | **store-setup** | Shopify プラン選定とテーマ調査は最初に始めないとブロッカーになる |
| 3 | **channel-sync-strategy** | 在庫同期方式の決定が遅れると移行・テストの全体が遅れる |

### 第2陣（Day 4〜7: 第1陣の初期成果を受けて始動）

| 順位 | エージェント | 理由 |
|:---:|---|---|
| 4 | **catalog-migration-planner** | 移行対象 SKU の選定基準は channel-sync の方針と連動する |
| 5 | **fulfillment-ops** | eBay 既存ノウハウの棚卸しは早期に着手できる |
| 6 | **price-auditor** | catalog-migration-planner の移行候補リストが出てから監査を開始する |

### 第3陣（Day 8〜: ストアの骨格が見えてから）

| 順位 | エージェント | 理由 |
|:---:|---|---|
| 7 | **growth-foundation** | ストア構造とコレクションが決まってから SEO・導線を設計する |
